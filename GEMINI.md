# 🍳 Railway 요리 비서 (Cooking Assistant) AI 가이드라인

본 프로젝트의 AI 어시스턴트로서 다음의 규칙과 제안 사항을 준수하여 개발을 지원합니다.

## 🛠️ 핵심 개발 가이드라인

### 1. 기술 스택 및 라이브러리 (Core Stack)
- **Frontend**: React 19, Vite, Tailwind CSS 4를 기본으로 사용합니다.
- **Styling**: 기본 Tailwind CSS를 사용하되, 고도의 커스텀이 필요한 경우 Vanilla CSS를 병행합니다.
- **Animation**: `framer-motion`을 적극적으로 사용하여 부드럽고 생동감 넘치는 인터랙션을 구현합니다.
- **Backend**: Node.js와 Express.js 환경에서 Prisma ORM을 사용하여 데이터베이스와 통신합니다.

### 2. UI/UX 디자인 원칙 (Premium Aesthetics)
- **프리미엄 지향**: 단순한 기능을 넘어 시각적으로 "와우(WOW)" 할 수 있는 세련되고 고급스러운 디자인을 추구합니다.
- **다크 모드 및 글라스모피즘**: 현대적인 색상 팔레트와 반투명 효과(Glassmorphism)를 활용하여 시각적 깊이를 더합니다.
- **마이크로 애니메이션**: 버튼 호버(hover), 로딩 상태, 카드 등장 시 세밀한 애니메이션을 추가하여 사용자 경험을 향상시킵니다.

### 3. 언어 및 현지화 (Language Policy)
- **한국어 우선**: 모든 사용자 인터페이스(UI)의 텍스트, 메시지, 도움말은 **한국어**를 기본으로 작성합니다.
- **주석 및 문서**: 서비스 운영 및 유지보수의 편의를 위해 코드 내 중요한 주석과 문서화 작업도 한국어를 지향합니다.
- **아티팩트 한국어화**: AI가 생성하는 모든 아티팩트(Implementation Plan, Walkthrough, Task 등)는 한국어를 기본으로 작성하여 사용자와의 소통 효율을 극대화합니다.

### 4. 코드 품질 및 아키텍처 (Code Quality)
- **클린 코드**: 가독성이 높고 유지보수가 용이한 코드를 작성합니다.
- **일관된 명명 규칙**: Prisma 스키마 필드명(예: `user_id`)과 프론트엔드 API 요청/응답 필드명을 일치시켜 정합성을 유지합니다.
- **컴포넌트화**: 재사용 가능한 원자적(Atomic) 컴포넌트 구조를 설계하여 개발 효율을 높입니다.

### 5. 데이터베이스 정합성 및 안전성 (Database Safety)
- **선제적 백업**: 스키마 변경, 데이터 이관, 대량 삭제 등 위험 요소가 있는 DB 작업 전에는 반드시 해당 데이터를 백업하거나 스냅샷을 생성하여 즉각적인 복구가 가능하도록 준비합니다.
- **Prisma 명명 규칙**: Prisma 스키마 필드명(`user_id` 등)과 프론트엔드 API 필드명을 일관되게 유지하여 정합성을 보장합니다.
- **마이그레이션 검토**: `prisma migrate dev` 실행 전 변경 사항이 기존 데이터에 미치는 영향을 충분히 검토합니다.

### 6. 버전 관리 및 동기화 (Versioning & Synchronization)
- **버전 동기화**: 앱 버전 업데이트 시 `package.json` (루트 및 프론트엔드), `PROJECT.md`, 그리고 기타 버전이 명시된 모든 파일의 버전을 상호 동기화하여 일관성을 유지합니다.
- **PROJECT.md 유지관리**: `PROJECT.md`는 항상 최신 변경 사항과 현재 앱 버전이 반영된 상태를 유지해야 하며, 버전이 오를 때마다 해당 내용을 업데이트합니다.
- **버전 이력 기록**: 버전 업데이트 시 프론트엔드의 `src/utils/version.ts` 파일에도 최신 버전(`APP_VERSION`, `LAST_UPDATED`)과 수정 내역(`VERSION_HISTORY`)을 반드시 추가하여 사용자 화면에 반영되게 합니다.
- **Git 커밋 메시지**: 수정 사항을 Git에 올릴 때 가급적 앱 버전을 커밋 메시지에 포함하거나 명시합니다.

### 7. 로컬 환경 검증 (Localhost Verification)
- **작업 후 즉시 검증 가능 상태**: 작업 수행을 마친 후에는 사용자가 즉각적으로 `localhost` 환경(예: `npm run dev`)에서 변경된 화면을 띄우고 테스트해 볼 수 있도록 정상 구동 가능한 상태를 보장하거나, 실행 안내를 제공해야 합니다.

### 8. 계획 우선 수립 (Plan First)
- **선 계획 후 실행**: 사용자가 새로운 기능 제안, 구조 변경, UI 개선 등의 지시를 내릴 경우, 코드를 직접 수정하기 전에 반드시 먼저 **구체적인 작업 계획서(Implementation Plan)**를 수립하여 사용자에게 제시하고 동의를 얻은 후 실제 개발 작업을 시작해야 합니다.

### 9. 데이터베이스 스키마 마이그레이션 (Prisma Migrate)
- **스키마 변경 후 필수 마이그레이션**: `schema.prisma` 파일이 수정되어 새로운 모델이 추가되거나 필드가 변경되는 등 데이터베이스 스키마 변동이 발생한 경우, 반드시 `npx prisma migrate dev` (개발 환경) 명령어를 실행하여 변경 사항을 실제 DB에 동기화해야 합니다. 마이그레이션 실행을 누락하여 코드가 런타임 오류를 일으키는 일이 없도록 주의합니다.

### 10. 가이드라인 (Rules) 동기화
- **Rules 추가 시 프로젝트 동기화**: 새로운 Rule이 명시되어 전역 가이드라인에 추가될 때, 반드시 프로젝트 폴더 내부에 있는 `GEMINI.md` 파일에도 동일한 내용을 반영하여 두 환경 간의 가이드라인을 항상 똑같이 동기화(Sync)해야 합니다.

## 💡 유용한 제안 (Assistant Suggestions)

- 새로운 기능을 제안할 때, 항상 **시각적인 향상 포인트**를 함께 고민합니다.
- 데이터베이스 마이그레이션이 필요한 경우, `prisma migrate dev` 실행에 대한 안내를 포함합니다.
- 복잡한 로직은 **Implementation Plan** 아티팩트를 통해 도식화하거나 단계별로 설명하여 사용자의 이해를 돕습니다.
